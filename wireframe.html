Architecture / wireframe

How the Valles Marineris 3000 app works

A single-file HTML app: one shell hosting 15 tab panels. Some tabs are narrative; some are live physics calculators that share data; two run real hydraulic engines and export .inp files. Below: the screen layout, the tab map, the data flow, and the engine/export layer.

1 · Screen layout (every tab shares this shell)

The chrome stays put; only the panel body swaps when you tap a tab.

3 · How the physics tabs connect

The resource tabs tell one coupled story — water makes air, both need power, and the engines move and collect the water. Arrows show the conceptual dependency a user follows.

4 · The engine & export layer (Model Scale tab)

Where the app stops being illustrative and produces real, openable hydraulic models.

The same EPANET/SWMM solvers that power the live calculator tabs are what the Model Scale tab scales up and serializes into .inp text — delivered by download, or by the copy-paste box when the sandbox blocks downloads.

5 · Shared machinery

Onboarding
Welcome modal on first load (sessionStorage flag) + a 6-step guided tour that scrolls to and highlights key controls. Re-openable via the ? button.
Render pattern
Each tab = controls + a render() function. Slider oninput → recompute numbers → rebuild metric cards → redraw inline SVG. No frameworks, no build step.
Single file
One self-contained .html: Tailwind via CDN, vanilla JS, hand-drawn SVG. No external data. Runs offline; deploys anywhere static.
